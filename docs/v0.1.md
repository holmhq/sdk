# @holmhq/sdk 0.1

`0.1.0` is the first public release of Holm's strict-TypeScript SDK. Its stable
scope is deliberately small: build browser and BFBB Holm apps with typed HTTP,
auth, uploads, immutable resources, and deterministic test seams. Preview and
reserved entry points are shipped so their boundaries can be exercised without
presenting unfinished runtimes as stable.

## Support contract

Stable entry points follow additive `0.1.x` compatibility: no breaking removal,
rename, or behavioral contract change within the minor line. A future removal
must be deprecated first. Preview and reserved entry points are outside that
freeze until explicitly promoted.

| Entry point | Status | Contract |
| --- | --- | --- |
| `@holmhq/sdk` | stable | Runtime-neutral core composition and shared types. |
| `@holmhq/sdk/core` | stable | Capabilities, caller identity, lifecycle, errors, diagnostics, cancellation, and wire values without DOM or Node ambient types. |
| `@holmhq/sdk/transports` | stable | Requests, responses, auth application, caller-partitioned caching, uploads, and credential-safe diagnostics. |
| `@holmhq/sdk/app` | stable | Holm app HTTP, auth, links, pagination, uploads, and built-in surface helpers. |
| `@holmhq/sdk/web` | stable | Browser runtime plus the `createWebApp()` convenience composition. |
| `@holmhq/sdk/state` | stable | Framework-neutral query, mutation, derived, history, and reconciliation resources. |
| `@holmhq/sdk/test` | stable | In-memory runtime and deterministic clock utilities. |
| `@holmhq/sdk/node` | preview | Node adapter and services; usable for bounded integrations, not frozen for `0.1.x`. |
| `@holmhq/sdk/sobek` | preview | Holm's injected server/Sobek adapter; not frozen for `0.1.x`. |
| `@holmhq/sdk/bridge` | reserved | Mailbox and service-slot contracts for tests and future native shells, not a production desktop/mobile runtime. |

See the full [capability matrix](capabilities.md) for what is deliberately not
available.

## Evidence behind the release

- The complete source, declaration-consumer, generated-artifact, BFBB, Vite,
  React, coverage, size, license, reproducibility, and package-content gates run
  through `npm run ci`.
- Stable declarations are checked against a frozen API manifest.
- Core is compiled separately without DOM or Node ambient libraries.
- The Sokoban pilot consumed a hash-verified vendored `dist/` tree, exercised
  auth, app HTTP, query and mutation resources, and ran end to end in Chromium
  against Holm `0.185.1`. The same build is live at
  [sokoban.zyt.app](https://sokoban.zyt.app).
- Holm-authority conformance was accepted against Holm `748cbe5`. The relevant
  `packages/holm-sdk` and `packages/holm-state` trees are unchanged at the live
  verification commit `afe4057f7ea98012d843717216675dd1a1043771`.

This evidence proves the bounded web release. It does not claim that preview,
reserved, or unavailable capabilities are production-ready.

## Distribution

Install from npm:

```bash
npm install @holmhq/sdk
```

BFBB apps can instead vendor the tracked ESM module tree. The tree is
content-hashed in `dist/manifest.json`; use an immutable Git SHA, the `v0.1.0`
tag, or the immutable npm version. See [Vendoring and updates](vendoring.md).
A deployed app should load its local copy rather than depend on a CDN at
runtime.

## Compatibility and runtime notes

- Package format: ESM JavaScript, TypeScript declarations, declaration maps, and
  source maps.
- Package engine: Node `>=20`; canonical release validation runs on Node 24.
- Web support is capability-based: ESM, Fetch-compatible request/response
  primitives, and browser URL/header primitives. Upload helpers additionally
  require the relevant file/blob primitives.
- Holm app wire behavior remains owned by Holm. The SDK does not invent a
  parallel protocol or bypass Holm authorization.
- Existing Holm `packages/holm-sdk` and `packages/holm-state` remain live during
  migration; this release does not delete or redirect them.

## Known integration edges

External sign-in/sign-out transitions are not observable by the SDK caller
provider automatically. Invalidate app HTTP cache and refresh auth-backed
resources after such a transition. Some Holm versions clear the session cookie
and then answer logout with a redirect whose followed request fails; re-check
session state instead of treating that transport error alone as proof that the
cookie survived. The shared vanilla/React example demonstrates this cleanup.

## Security boundary

Transport diagnostics redact structurally marked sensitive URL, path, query,
header, and body material. Callers remain responsible for marking signed URLs,
magic links, invite paths, and query tokens with the request sensitivity
metadata. Never publish credentials, private payloads, or full sensitive logs.
Report suspected integrity or credential-redaction defects privately to the
Holm HQ maintainers.
