# v0.1-web private RC contract

This document is the scoped documentation and metadata contract for the private
`0.1.0-rc.1` code/artifact checkpoint. It is for local owner handoff and
immutable vendoring readiness only. This checkpoint is not an npm publication, tag, GitHub release, deployment, production proof, pilot result, browser/vendor soak claim, or promotion to `0.1.0`.

The package remains private. Deployed apps must not consume a mutable branch,
release workflow, runtime CDN dependency, or unpublished npm package. A real app/browser soak and owner-present promotion to `0.1.0` are future gates after
this RC stops. Issue `#015` remains open; this document does not claim the broad
README/API/framework/migration closeout is complete.

## Support matrix

Stable `0.1.x` compatibility: no breaking removal, rename, or behavioral contract change; additive changes only; deprecate before any future removal.
Preview and reserved imports are exempt from the stable `0.1.x` freeze and may
change before they are promoted by a future owner-approved plan.

| Entry point or surface | Status | v0.1-web contract |
| --- | --- | --- |
| `@holmhq/sdk` | stable | Root package export; environment-neutral stable surface for approved v0.1-web SDK composition. |
| `@holmhq/sdk/core` | stable | Runtime-neutral core contracts; no DOM or Node ambient type leakage. |
| `@holmhq/sdk/transports` | stable | Transport request/response/cache/sensitivity contracts; credentials stay adapter-private and redacted by contract. |
| `@holmhq/sdk/app` | stable | Adopted app/auth/link/pagination/upload/surface helper contracts. Holm app wire behavior remains GET/POST. |
| `@holmhq/sdk/web` | stable | Browser/BFBB web runtime and convenience composition for the capability-based web baseline. BFBB authored root `index.html` takes precedence over generated presentation. |
| `@holmhq/sdk/state` | stable | Framework-neutral resource/query/mutation/derived-resource contracts and immutable snapshots. |
| `@holmhq/sdk/test` | stable | Deterministic conformance/test seams for SDK consumers and adapters. |
| `@holmhq/sdk/node` | preview | Shipped and tested for bounded adapter composition, but Node 20/22/24 behavior and CLI ergonomics are not frozen. |
| `@holmhq/sdk/sobek` | preview | Shipped and tested for Sobek adapter composition, but server-runtime behavior is not frozen. |

## Supported imports and BFBB usage

Stable v0.1-web imports are limited to:

```text
@holmhq/sdk
@holmhq/sdk/core
@holmhq/sdk/transports
@holmhq/sdk/app
@holmhq/sdk/web
@holmhq/sdk/state
@holmhq/sdk/test
```

Preview imports are `@holmhq/sdk/node` and `@holmhq/sdk/sobek`. Reserved import
`@holmhq/sdk/bridge` exists for mailbox/mock boundaries only. Do not present any
preview or reserved import as stable application support.

Raw BFBB apps should vendor the tracked ESM files from an immutable SDK commit
or reviewed tag and then import local files at runtime:

```bash
SDK_REF=<immutable-commit-sha-or-reviewed-tag>
mkdir -p vendor/holm-sdk
curl -fL "https://cdn.jsdelivr.net/gh/holmhq/sdk@${SDK_REF}/dist/holm.js" \
  -o vendor/holm-sdk/holm.js
curl -fL "https://cdn.jsdelivr.net/gh/holmhq/sdk@${SDK_REF}/dist/manifest.json" \
  -o vendor/holm-sdk/manifest.json
```

```js
import { createHolm } from './vendor/holm-sdk/holm.js'
```

Use an immutable Git SHA or reviewed tag for every vendored update. Never use
`@main` for deployed apps. The CDN URL is only a preparation source for copying
files into the app; the deployed BFBB app should serve its local vendored copy.

## Hash verification

Record the SDK version, source ref, artifact path, and SHA-256 from
`dist/manifest.json` next to the vendored files. Verify copied artifacts before
committing an app update:

```bash
# Example: replace the hash with the dist/manifest.json sha256 for holm.js.
printf '%s  %s\n' '<sha256-from-dist-manifest>' 'vendor/holm-sdk/holm.js' \
  > vendor/holm-sdk/holm.js.sha256
sha256sum -c vendor/holm-sdk/holm.js.sha256
```

If the hash check fails, discard the copied file and fetch again from the same
immutable ref. Do not switch to `@main`, a branch name, or a different ref to
make verification pass.

## Update workflow

1. Choose an immutable SDK commit SHA or owner-reviewed tag. Do not use `@main`.
2. Review the RC support status and compatibility notes for the target ref.
3. Copy the required `dist/` artifacts into the app's vendor directory.
4. Copy or record the matching `dist/manifest.json` metadata.
5. Run `sha256sum -c` for each vendored artifact.
6. Run the app's normal BFBB/static checks and any package-manager build checks.
7. Commit the app's vendored files, checksum metadata, and source ref together.

## Rollback workflow

Rollback means restoring the previously vendored SDK files and their recorded checksum metadata. Revert the app commit that changed the vendored SDK files, or
copy back the prior vendor directory and prior SHA-256 records, then run
`sha256sum -c` before redeploying the app. A rollback must not fetch `@main`,
move to a different unreviewed ref, or change app behavior to fit a preview or
reserved SDK surface.

## Security notes

Report suspected SDK integrity or credential-redaction issues privately through
Holm HQ's security contact or the owner-approved private channel for this RC.
Do not paste credentials, signed URLs, invite links, private payloads, or full
sensitive logs into public issues. Transport diagnostics redact structurally
marked sensitive URL, path, query, header, and body material, but callers remain
responsible for marking URL-borne credentials with the SDK sensitivity metadata.

## Protocol authority and future gates

Holm remains the runtime and protocol authority. The RC preserves Holm's app
contract direction: `GET` observes resources or queries state, `POST` performs
commands or state transitions, and BFBB authored root `index.html` takes
precedence over generated presentation. This SDK document must not redefine
Holm's app wire contract, claim generated UI ownership, or make desktop/mobile,
realtime, collaboration, framework binding, admin, generated CLI, or arbitrary
SSR promises.

The private `0.1.0-rc.1` checkpoint stops before real-app pilot evidence,
browser/vendor soak results, public release notes, npm publication, tags,
GitHub releases, deployments, credentials, cloud/production mutation, and
promotion to `0.1.0`.
